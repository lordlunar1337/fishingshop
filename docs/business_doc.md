# Бізнес-документ: Магазин рибальської снаряги "FishHook Shop"

## 1. Назва системи та короткий опис

"FishHook Shop" — це веб-застосунок для онлайн-продажу рибальської снаряги: вудилищ, котушок, жилок, гачків, приманок, аксесуарів та супутніх товарів. Система дозволяє користувачам переглядати каталог товарів, фільтрувати та сортувати їх, додавати обрані позиції в кошик та оформлювати замовлення онлайн.

Основна мета системи — зробити купівлю рибальських товарів простою та зручною, особливо для користувачів, які вже знають, що їм потрібно, і хочуть швидко знайти потрібну снарягу без походів у офлайн-магазин. Для власників магазину система дає можливість керувати асортиментом, цінами, залишками на складі та замовленнями клієнтів.

## 2. Користувачі (ролі)

- Гість

  - може переглядати каталог товарів
  - може фільтрувати та сортувати товари
  - може додавати товари в кошик, але для оформлення замовлення потрібно зареєструватися або увійти

- Зареєстрований покупець

  - має всі можливості гостя
  - може оформляти замовлення
  - може переглядати історію своїх замовлень
  - може редагувати свій профіль та контактні дані

- Адміністратор

  - керує каталогом товарів (додавання, редагування, видалення)
  - керує категоріями товарів
  - переглядає та змінює статуси замовлень
  - керує користувачами (блокування, надання ролей)

- Менеджер складу
  - переглядає товари та їхні залишки
  - оновлює інформацію про наявність товарів
  - має доступ до списку замовлень для підготовки відправки

## 3. Основні сценарії використання

1. Гість відкриває сайт, переглядає каталог товарів, користується фільтрами за категоріями (вудилища, приманки, котушки), дивиться деталі конкретного товару.
2. Користувач додає кілька товарів у кошик, переходить у кошик, коригує кількість, оформляє замовлення, вказуючи контактні дані та спосіб доставки.
3. Зареєстрований покупець заходить у свій профіль, переглядає історію замовлень, відкриває деталі конкретного замовлення та повторює його.
4. Адміністратор заходить у панель керування, додає новий товар (вудилище з конкретними характеристиками), встановлює ціну, категорію та опис.
5. Менеджер складу переглядає список нових замовлень, готує відправку, відмічає замовлення як "відправлене".

## 4. Функціональні вимоги

- Система повинна надавати каталог товарів із можливістю:

  - перегляду списку товарів
  - перегляду детальної сторінки товару
  - фільтрації за категоріями
  - пошуку за назвою

- Система повинна надавати кошик:

  - додавання товару в кошик
  - перегляд товарів у кошику
  - зміна кількості або видалення товару з кошика

- Система повинна дозволяти оформлення замовлення:

  - заповнення контактних даних
  - вибір способу доставки (наприклад, Нова Пошта)
  - збереження замовлення в базі даних

- Система повинна підтримувати реєстрацію та авторизацію користувачів:

  - реєстрація нового користувача
  - вхід із використанням e-mail та пароля
  - вихід із системи

- Система повинна мати адміністративну панель:
  - додавання, редагування, видалення товарів
  - керування категоріями
  - перегляд списку замовлень та зміна їх статусу

## 5. Нефункціональні вимоги

- Швидкодія

  - сторінка каталогу з базовим фільтром повинна завантажуватися не довше 2–3 секунд при нормальному навантаженні
  - API-запити до бекенду повинні оброблятися в межах 300–800 мс у типових сценаріях

- Безпека

  - паролі користувачів повинні зберігатися у вигляді хешів
  - доступ до адміністративної панелі повинен бути обмежений роллю адміністратора
  - усі основні операції авторизації повинні виконуватися через захищене з’єднання (HTTPS у продакшн-середовищі)

- Зручність використання

  - інтерфейс має бути адаптований для десктопів і планшетів
  - мінімум кроків для оформлення замовлення
  - зрозуміла навігація між категоріями товарів

- Надійність

  - система повинна коректно обробляти помилки (недоступність БД, помилки мережі)
  - у випадку збоїв користувач повинен отримувати зрозумілі повідомлення про помилки

- Масштабованість
  - архітектура повинна дозволяти у майбутньому додати окремі сервіси (наприклад, окремий сервіс авторизації або платежів) без повної переробки системи

## 6. Обмеження

- На початковому етапі система не буде мати:

  - нативних мобільних застосунків (тільки веб-версія)
  - інтеграції з реальними платіжними системами (оплата може моделюватися як "оплата при отриманні")
  - складної системи знижок, купонів та промокодів
  - інтеграції з кількома мовами інтерфейсу (початково тільки українська)

- Система не буде виконувати:
  - ведення бухгалтерського обліку
  - повну автоматизацію логістики та відстеження посилок
  - управління кількома магазинами або брендами в рамках однієї інсталяції

## 7. Архітектурні блоки (пояснення)

- Client (Frontend)

  - React-застосунок, який працює у браузері користувача
  - відповідає за відображення каталогу, кошика, форми замовлення, профілю користувача
  - спілкується з бекендом через HTTP-запити (REST API)

- Backend (API)

  - Node.js + Express
  - реалізує бізнес-логіку: робота з товарами, кошиком, замовленнями, користувачами
  - виконує авторизацію та перевірку прав доступу
  - взаємодіє з базою даних

- Database

  - на початковому етапі SQLite для прототипу
  - зберігає товари, категорії, користувачів, замовлення
  - у майбутньому може бути замінена на PostgreSQL або іншу реляційну БД

- Auth Module (майбутній модуль)

  - окремий логічний блок у бекенді, який відповідає за реєстрацію, вхід, вихід, перевірку токенів
  - може бути виділений у окремий сервіс у майбутньому

- Notifications Module (майбутній модуль)

  - надсилання e-mail або інших повідомлень про створення та статус замовлення
  - на етапі курсової може бути реалізований як логування або фейкові нотифікації

- Payments Module (майбутній модуль)
  - інтеграція з платіжними системами
  - на етапі курсової може бути лише зафіксований як майбутній блок без реалізації
