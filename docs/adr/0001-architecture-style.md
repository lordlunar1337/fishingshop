# ADR 0001: Вибір архітектурного стилю

- Статус: Accepted
- Дата: 2025-12-02

## Контекст

Проєкт: веб-застосунок "FishHook Shop" для онлайн-продажу рибальської снаряги.

Команда працює в навчальному середовищі, обмежений час семестру. Невелика кількість розробників, прості процеси деплою та відсутність вимог до масштабування на десятки незалежних команд.

На поточному етапі функціонал обмежується одним застосунком з каталогом товарів, замовленнями, оплатами та доставкою.

## Рішення

Обрано стиль **модульний моноліт**.

- Єдиний бекенд-застосунок на Node.js + Express.
- Усередині бекенду система ділиться на модулі за доменними контекстами (Catalog, Ordering, Payments, Shipping, Users) та на шари (api, service, domain).
- Доменні моделі зберігаються в окремій папці `domain/` і можуть перевикористовуватись різними модулями.

## Альтернативи

1. **Мікросервіси на пілоті**

   - Плюси: можливість незалежного масштабування сервісів у майбутньому, чіткі межі контекстів на рівні деплою.
   - Мінуси: значно більша складність на етапі навчального проєкту (деплой, спостереження, тестування, комунікація між сервісами, розподілена транзакційність).

2. **Класичний моноліт без модульної структури**
   - Плюси: найпростіша реалізація.
   - Мінуси: складно підтримувати при розростанні проєкту; важко виділяти чіткі bounded context та дотримуватись DDD.

## Наслідки

- Для розробки:

  - Спрощена інфраструктура, один застосунок бекенду.
  - Можливість поступово виділяти внутрішні модулі й шари без зміни моделі деплою.
  - Легше пояснювати і демонструвати архітектуру в рамках курсової.

- Для релізів:
  - Один артефакт для деплою бекенду.
  - Немає потреби в оркестрації кількох сервісів.

## Умови перегляду

До питання мікросервісів можна повернутися пізніше, якщо:

- зʼявляться окремі команди, що відповідають за різні bounded context;
- з’явиться вимога незалежно масштабувати окремі частини системи;
- кількість функціоналу та навантаження суттєво зростуть.
